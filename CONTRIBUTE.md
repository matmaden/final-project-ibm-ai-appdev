## Style Guide and Coding Standards

To maintain a clean codebase, please ensure your code follows these standards:
* **Code Formatting:** Run the linter/formatter before committing (e.g., `npm run lint` or `black .`).
* **Testing:** Ensure all existing tests pass and write new tests for your changes. Run tests using `[Your Test Command]`.
* **Commit Messages:** Write concise, clear commit messages. Use the imperative mood (e.g., "Fix user login bug", not "Fixed user login bug").

## Pull Request Process

When you are ready to submit your changes, follow this process:

1. **Push your branch** to your forked repository:
   ```bash
   git push origin feature/your-feature-name
   ```
2. **Open a Pull Request (PR)** against our `main` (or `development`) branch.
3. **Fill out the PR template** completely, describing what changes were made and why.
4. **Link relevant issues** in the PR description (e.g., `Closes #123`).
5. **Await review** from project maintainers. Be prepared to make modifications if requested.

## Reporting Bugs

Before creating a new bug report, please search our existing open issues to see if it has already been reported. 

If it hasn't, please open an issue and include:
* A clear, descriptive title.
* Steps to reproduce the issue.
* Expected vs. actual behavior.
* Relevant screenshots, logs, or error messages.
* Your environment details (OS, runtime version, browser version, etc.).
